# 11 - Load Balancer, DNS, and TLS

Only the API gateway should be public.

## Public entry point

Expose this service:

```text
api-gateway
```

Do not expose these publicly:

```text
auth-service
order-service
payment-service
notification-worker
```

## AWS Load Balancer Controller

Install AWS Load Balancer Controller in EKS.

It creates an AWS Application Load Balancer when you create an Ingress resource.

## ALB name pattern

The final ALB name may be generated by Kubernetes/AWS, but use tags and ingress annotations to identify it:

```text
Project=edfp
Environment=dev
Component=api-gateway
```

## DNS with Route 53

Optional but professional.

Example domain records:

```text
api-dev.yourdomain.com
api-staging.yourdomain.com
api.yourdomain.com
```

Point the DNS record to the ALB.

## TLS certificate

Use AWS Certificate Manager for TLS certificates.

Example certificate names/tags:

```text
edfp-dev-api-certificate
```

Recommended:

```text
HTTPS only
HTTP redirects to HTTPS
```

## Routing design

Public routes:

```text
/api/register -> api-gateway -> auth-service
/api/login -> api-gateway -> auth-service
/api/orders -> api-gateway -> order-service
/api/payments -> api-gateway -> payment-service
```

Internal services communicate through Kubernetes service DNS.

## Verification checklist

- only api-gateway has public ingress
- ALB is created successfully
- `/health` works through public URL
- `/ready` works internally or through protected route
- HTTP redirects to HTTPS if TLS is configured
- internal services are not publicly reachable
